# TurboSearch 性能改进计划

## 概述

本文档定义了 TurboSearch 项目的性能优化方案，包括识别出的瓶颈、改进策略和实施步骤。

---

## 阶段一：搜索性能优化

### 1.1 搜索缓存机制
**目标**: 避免重复搜索相同关键词

**当前状态**:
- ✅ 实现搜索结果缓存 (LRU + Arc)
- ✅ 使用 LRU 缓存策略，限制缓存条目数量
- ✅ 缓存 key = pattern + case_sensitive + regex + glob + index_key
- ✅ 缓存过期机制（CACHE_VERSION AtomicU64）

**预期效果**: 重复搜索速度提升 10-100 倍

### 1.2 模糊搜索早期终止
**目标**: 减少不必要的模糊匹配计算

**当前状态**:
- ✅ 实现自适应早期终止 (5000/50000 阈值)
- ✅ 当找到 `limit * 10` 个高分结果时停止搜索
- ✅ 使用 `select_nth_unstable` 实现高效的部分排序

**预期效果**: 大索引搜索速度提升 30-50%

### 1.3 并行阈值自适应
**目标**: 根据索引大小自动选择最优并行策略

**当前状态**:
- ✅ 硬编码阈值 5000/50000 自适应

**改进措施**:
- [ ] 实现自适应阈值计算
- [ ] 根据 CPU 核心数动态调整并行度
- [ ] 使用 `rayon::ThreadPool` 复用线程

## 阶段二：索引性能优化

### 2.1 启用 jwalk 排序
**目标**: 利用 jwalk 的内置并行排序优势

**当前状态**:
- ✅ `sort(true)` 启用排序
- ✅ `skip_hidden(true)` 跳过隐藏文件

**预期效果**: 索引速度提升 20-40%

### 2.2 增量索引集成
**目标**: 实现文件变化的实时更新

**当前状态**:
- ✅ FileWatcher 已集成到 GUI
- ✅ 实现增量索引更新逻辑 (Created/Modified/Removed)
- ✅ Browse/Re-index 时自动停止/启动 Watcher

**改进措施**:
- [x] 将 FileWatcher 集成到 GUI
- [x] 实现增量索引更新逻辑
- [x] 添加索引更新事件通知

### 2.3 MFT 快速索引
**目标**: 利用 Windows NTFS MFT 实现超快初始化

**当前状态**:
- `mft_reader.rs` 已实现 (使用 FindFirstFileW API)
- jwalk 索引已高度优化 (并行 + 排序)

**说明**:
- True MFT 解析需要理解 NTFS MFT 结构和 FILE_RECORD_SEGMENT (复杂度高)
- jwalk 索引已使用并行处理，排序已启用

**预期效果**: jwalk 已实现秒级索引

## 阶段三：GUI 性能优化

### 3.1 真正的 LRU 缓存
**目标**: 优化预览缓存内存使用

**当前状态**:
- ✅ 使用 LruCache，100 条目限制
- ✅ 基于访问频率的淘汰策略

**改进措施**:
- [x] 实现 `LruCache` 结构
- [x] 基于访问频率和时间的淘汰策略

### 3.2 异步文件预览
**目标**: 避免预览加载阻塞 UI

**当前状态**:
- ✅ 使用 `preview_channel` 后台线程加载

- [ ] 使用 tokio 或 async 异步读取
- [ ] 实现预览加载状态指示
- [ ] 添加取消预览加载能力

### 3.3 清理重复代码
**目标**: 减少维护成本和潜在 bug

**当前状态**:
- settings dialog 在两处定义

**改进措施**:
- [ ] 合并重复的 settings dialog 代码
- [ ] 统一 UI 组件管理

## 阶段四：内存和并发优化

### 4.1 字符串优化
**目标**: 减少内存分配

**当前状态**:
- 大量 `to_lowercase()` 调用

**改进措施**:
- [ ] 复用 `name_lower` 字段
- [ ] 使用 `std::borrow::Cow` 减少复制

### 4.2 并发锁优化
**目标**: 减少锁竞争

**当前状态**:
- 使用标准库锁

**改进措施**:
- [ ] 评估使用 `parking_lot` 替代
- [ ] 使用 `dashmap` 的更高效模式
- [ ] 分析并优化热点锁

### 4.3 正则缓存优化
**目标**: 更智能的正则表达式缓存

**当前状态**:
- 简单的数量限制清理

**改进措施**:
- [ ] 实现 LRU 淘汰策略
- [ ] 添加缓存命中率统计
- [ ] 动态调整缓存大小

## 验证方法

1. **性能基准测试**:
   - 1000 文件索引 + 搜索
   - 100000 文件索引 + 搜索
   - 1000000 文件索引（仅索引）

2. **内存分析**:
   - 使用 `valgrind` 或 `tracy` 检测内存泄漏
   - 监测缓存内存使用

3. **UI 响应测试**:
   - 搜索时 UI 是否卡顿
   - 预览加载是否流畅
